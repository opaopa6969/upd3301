# upd3301 vs M88 — 挙動比較

`upd3301` が **M88**（`bubio/M88M`）とどう違うかの生きた記録。[`../m88ref/`](../m88ref/)
のヘッドレス参照神託で作る。手法: 同じ `.d88` を両方でヘッドレス・同フレーム数起動し、
fingerprint を比較する ── 最終 `E6CD`（ゲーム固有フラグ。多くのタイトルでキーボード
スキャンの門）、非ゼロテキストVRAMバイト数、ディスク読み活動。差分は「手がかり」で
あって「判決」ではない。各差分は命令単位で追う（テキストウィンドウのバグもこうやって
見つけた ── [io-ports](./io-ports.ja.md) と捜査記録参照）。

修正（port 70h）が大きな梃子だった。`E6CD == 0xff` は必ずしも「停止」ではない: 数本
（Again/Eldrad/D-SIDE/Aggres）は M88 でも 0xff を保つ。

### この表を読むときの注意
- **読み数はまだ同一単位で比較できてない。** うちは `READ DATA` **コマンド**（FDC op
  0x06）、refdrv の `g_rdN` は `FDC::ReadData` 呼び出し数（セクタ単位かも）。両者が同じ
  イベントを数えるまで生の数は「大まかな進捗指標」。
- **結果ヘッダの一致 ≠ ペイロードの一致。** `C/H/R/N` 結果バイトが連続で一致しても、それは
  *番地*が合っただけで、転送された256バイトの**データ**が合った保証はない。差分を追うときは
  結果ヘッダ7バイトだけでなく、read毎のペイロードを両側でダンプして diff すること（FDCが
  返すバイトをフック）── でないと測定器がデータ差を隠す。*(Dragon Buster の原因ではなかった
  ── あれは2枚組ハーネスのアーティファクト ── が、次の1本のための正しい作法。credit: codex。)*

## 既知の差分（追うべき手がかり）

1. **Dragon Buster — 解決: 2枚組ゲーム、スイープが1枚しか挿してなかった。**
   `Dragonb.d88` は**2イメージ ── "DISK A" と "DISK B"**。タイトルは DISK B を
   **ドライブ1（unit1）**から読む。DISK A しか挿してないと空ドライブを読んで
   `ST0=0x45`（`_rwError`, unit1にディスク無し）→ SEEK→SENSE→READ を永久ループ。両方
   挿す（DISK A→drive0, DISK B→drive1）と正常: **reads 22265 → 21**、ループ消滅。実
   フロントエンド（`demo/machine.html` `ingestDisks`）は image0→drive0, image1→drive1 を
   自動割り当てするので2枚組ゲームは UI で"just work"する ── 1枚しか挿してなかったのは
   比較*ハーネス*の方だった。エミュのバグではない。*(codex が2ドライブ仮説を指摘。)* **下の
   スイープ手法に修正を反映。**
2. **Ys1 / Abyss2 — f250で `E6CD` 差 ── バグではない（起動後の位相差）。** 両者とも
   E6CD は~200フレーム `0`（＝起動してゲームプレイ到達）、その後タイトル固有値へ遷移
   （Ys1→fc, Abyss2→1）する時刻が M88 と数フレームずれるだけ。f250 スナップショットが
   *起動後*実行の別の点を捉えただけ ── タイミング/進捗の位相であって欠陥ではない。*(解決。)*
3. **177 — tvNZ が M88 より多い ── バグではない（起動後の位相差）。** 同様: E6CD は早期に
   `0`、その後 `0x24`。タイトルは動いてる。tvNZ 差はゲームプレイで数フレームずれてるだけ。*(解決。)*
4. **Z80 未定義フラグ（X/Y, bit 3/5）── 実装済み・systematicな根ではない。** `z80.js` は
   主要ALU演算（`_add/_sub/_inc/_dec` と AND/OR/XOR の `SZP` テーブル）で結果から F3/F5 を
   立てる。軽井沢ロード中に観測した X/Y 不一致は無害な経路での被演算差で、フラグバグでは
   ない。残る既知ギャップ: **block-IO（LDIR/CPIR…）の未定義フラグは近似**。依存タイトルが
   出たら監査。*(低優先。)*

まとめ: スイープと追試の結果、このセットに**残る挙動差はゼロ**。各タイトルは M88 と一致
するか、起動後のタイミング位相だけの差か、（Dragon Buster）ハーネスのアーティファクト
だった。実バグはテキストウィンドウ1件のみ。

## バッチスイープ（353本・250フレーム）

`tools/batch-compare.mjs` は `.d88` コレクション全体を両エミュに通し、
E6CD/tvramNZ の指紋を diff して、ミスマッチを **画面充填量（tvramNZ）が
一致するか**で振り分ける。単一フレームのE6CDスナップショットはノイジー
（ゲーム固有のフラグバイトを各エミュで別のアニメフレームで捕まえる）なので、
「起動はするがE6CDが違う」は**バグではない**よくあるケース。

| カテゴリ | 本数 | % | 意味 |
|----------|------|---|------|
| E6CD完全一致 | 304 | 86% | 250fで同じ指紋 |
| 位相ノイズ（両方起動・tvNZ±15%内） | 31 | 9% | 両方で動作。スナップショットが別フレーム — **バグではない** |
| 本物の乖離リード（tvNZ>15%差） | 16 | 4.5% | 画面内容が実際に違う — 追う価値あり |
| ブランク（両方tvNZ<200） | 1 | — | CHOPLIFT（要キー入力/フレーム増） |
| refdrvエラー | 1 | — | M88自身がリトルコンピューターピープルを起動失敗 |

**→ 335/353（95%）がM88を追従**（完全一致＋位相ノイズ）。位相ノイズには
解決済みの Ys1 / Abyss2 / Sorcerian が含まれ、同一タイトルの重複コピー
（Rayieza≡地球戦士ライーザ、Hajya≡覇者の封印）は*同じ*乖離に着地する —
指標は安定している。

**サイクル精度問題への読み方**：もしフレーム級タイミングが律速なら乖離は
広範に出て位相ノイズ層が実バグになるはず。実際は95%がフレーム粒度で追従 —
フレームステップのコアで大半には十分。16本のリードは**タイトル固有**で
系統的なタイミングの壁ではないので、サイクル完全の全面書き直し（動いている
95%を壊すリスクもある）より個別に追う方が良い。

**本物の乖離リード（250fで画面内容が違う）：**

```
ﾄﾘﾄｰﾝ            M88 tv0(空)     ours tv4096(満)  — 反転。graphics-only?
Skyfox           M88 tv2678      ours tv643       — oursが手前で停止
Snatcher         M88 tv2678      ours tv1173
Makaimura        M88 tv3126      ours tv1987
tennis           M88 tv2678      ours tv4096(満)  — oursがゴミ埋めの可能性
GAZZEL           M88 tv3077      ours tv2048
Deringer / Stercru / starclsr / ROLLER / キャッスルエクセレント
Rayieza(≡地球戦士ライーザ) / Hajya(≡覇者の封印) / ロリータシンドローム
```

**ハーネスの落とし穴（重要）**：our側は4つのN88拡張ROM（`n88_0..3.rom`）を
`ext`として 6000-7FFF にマップして構築する必要がある — この拡張ROMこそが
N88-DISK-BASIC。省くとマシンはN88-BASICプロンプトに落ちて**どのゲームも
起動しない**のに、全タイトルがE6CD=00で偽の「一致」になる。このスイープの
初回はまさにこれを踏んで無意味な83%を報告した。`ext`を配線して上の本当の
姿が出た。

## 再現/拡張手順

```sh
# M88側（一度ビルド）:
m88ref/build.sh
m88ref/_m88m_build/M88M/refdrv <romDir> <disk.d88> 250   # 最終 E6CD, tvramNZ, g_rdN を出力

# コレクション全体を両エミュで、分類付き:
node tools/batch-compare.mjs <romDir> <diskDir> 250

# うち側（batch-compareが1本ごとにやること）: Nodeハーネス ──
#   new Pc8801Machine({main, ext, sub, mode:'n88'}) → insertDisk → stepFrame ×250、
#   その後 m.ram[0xe6cd] を読み、非ゼロ m.tvram[] を数え、globalThis.__fdcCmd をフック。
# 重要#1: `ext` = 4つのN88拡張ROMを連結（n88_0..3.rom を i*0x2000、6000-7FFFにマップ）。
#   このROMこそN88-DISK-BASIC。無いとディスクが起動せず全タイトルがE6CD=00で偽一致。
# 重要#2: 複数枚 .d88 は *全イメージ* を挿すこと ──
#   const d = parseD88All(bytes); d.forEach((img,u) => u<2 && m.insertDisk(u,img));
# 2枚組タイトルで image0 しか挿さないと空のドライブ1でループする
# （それが Dragon Buster の "差分" の正体だった）。
```
完全な手法と、貼るだけのサブエージェント prompt は [`../m88ref/README.ja.md`](../m88ref/README.ja.md)。
見つけた行/差分はここに追記していく。
