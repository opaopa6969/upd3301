# エミュレータのホスティング — 公開クライアント専用 vs 認証付きper-user ROM

PC-8801 エミュレータを web でどうホスティングするかの設計。エミュは**純クライア
ント JS** — [`z80.js`](../z80.js)・[`machine88.js`](../machine88.js)・
[`ym2203.js`](../ym2203.js)・[`ym2608.js`](../ym2608.js)・[`crt.js`](../crt.js)
ほか全部がブラウザ内で動く（[`demo/machine.html`](../demo/machine.html)・
[`demo/opn-scope.html`](../demo/opn-scope.html)）。サーバ側エミュレーションは無く、
NEC 著作物もリポジトリに一切入っていない。これがホスティング全体の形を決める：
**面白い問いは「CPU がどこで動くか」（タブ内で動く）ではなく「ROM がどこに在る
か」だ。**

これは2つのホスティング層の設計契約であり、既存コード
（[`demo/romstore.js`](../demo/romstore.js)）と我々がデプロイする基盤（volta）に
即して書く。何を・どの順で作り・どう検証するかを言う。未確認の点は
**【要確認】** と明示する。

## 1. ROM 問題が形を決める

マシンは NEC の **N-BASIC / N88-BASIC** ROM から起動し、ゲームの **`.d88` ディスク
イメージ** を動かす。全部著作物（NEC 分は 2049 年まで、ゲーム publisher 分はもっと
長い）。**我々はそれを一切配布しない。** ユーザが自分の吸い出しを持ち込む —
BYO-ROM。だから効く設計判断はただ一つ：ユーザがバイトを供給した後、*そのバイトは
セッション間どこで休むのか？*

答えは2つ、層も2つ：

| | **3301.unlaxer.org**（公開） | **emulator.unlaxer.org**（認証付き） |
|---|---|---|
| 認証 | 無し | volta ログイン必須 |
| ROM の在り処 | このブラウザの IndexedDB | (Phase 1) 同左＋gate ・ (Phase 2) **サーバ側の自分のフォルダ** |
| バイトは端末から出る？ | **絶対に出ない** | *自分専用* の暗号化 per-user ストアにのみ |
| 用途 | 「この端末で試す」 | 「一回上げれば全端末で使える」 |
| 状態 | **実装済**（Phase 0） | **これから作る**（Phase 1 → 2） |

## 2. アーキテクチャ

```
                        Cloudflare (ワイルドカード *.unlaxer.org, TLS)
                                       │
                 ┌─────────────────────┴──────────────────────┐
                 │                                             │
        3301.unlaxer.org                             emulator.unlaxer.org
        (公開層)                                     (認証層)
                 │                                             │
        静的ファイル, no-cache                        volta-gateway (:80)
        (serve.py / 任意の静的ホスト)                 route: app_id 指定 → 認証
                 │                                    ┌────────┴─────────┐
                 ▼                                    │ AuthChecked      │
        ┌──────────────────┐                         │  ← volta /verify │
        │  ブラウザタブ    │                         └────────┬─────────┘
        │  ┌────────────┐  │                              (認証済み)
        │  │ machine88  │  │                                  │
        │  │ z80 / OPN  │  │                                  ▼
        │  └─────┬──────┘  │                         ┌──────────────────┐
        │   romstore.js    │                         │  ブラウザタブ    │
        │        │         │                         │  romstore.js     │
        │        ▼         │                         │    │      │      │
        │  ┌────────────┐  │                         │  local    remote │
        │  │ IndexedDB  │  │  ← ROM はタブから       │ IndexedDB  同期  │
        │  │ (このPC)   │  │    絶対に出ない         │ (キャッシュ)│    │
        │  └────────────┘  │                         └───────────┼─────┘
        └──────────────────┘                                     ▼
                                                        per-user ROM ストア
                                                      (オブジェクトストレージ/ファイル,
                                                       暗号化, オーナー鍵付き)
```

公開層は今そのまま出荷されているエミュ：Cloudflare の裏の静的ファイル、ROM は訪問者
自身の IndexedDB にのみ在る。認証層は *同じ静的アプリ* を volta-gateway の裏に置いて
「誰か」を分かるようにしたもの＋per-user サーバストア。クライアントはそのストアを
**任意の同期先** として扱い、正典（source of truth）とはしない。

## 3. Phase 0 — 公開クライアント専用（実装済）

これは既に動いていて、基準点。ここに願望は無い。

- **ホスティング**：静的ファイル。[`serve.py`](../serve.py) が
  `Cache-Control: no-cache` を送り、Cloudflare がデプロイ後の古いモジュールを配らず
  再検証する。`3301.unlaxer.org` は CF の裏のこのディレクトリそのもの。
- **永続化**：[`demo/romstore.js`](../demo/romstore.js) がアップロードされた各 blob を
  IndexedDB に持つ（`DB 'upd3301-roms'`・store `blobs`・`keyPath: 'role'`）。現状の
  role は `rom`（起動 ROM）・`font`（CGROM）・`disk`（`.d88`）。API は
  `putRom / getRom / listRoms / clearRoms`。
- **自動起動**：[`machine.html`](../demo/machine.html) のファイル `onchange` ハンドラ
  が選択後に `putRom(...)` を呼び、ロード時の `restoreFromStore()` が
  `font`/`disk`/`rom` を読み戻して起動する — **「一度アップ」であって「我々にアップ」
  ではない。** UI も *「保存済み(このブラウザのみ)」* と言っている。
- **プライバシー不変条件**：バイトはタブから出ない。アップロード endpoint が存在しな
  い。サーバを足すときに壊してはいけないのがこの性質。

## 4. なぜクライアント優先が既定か（そしていつサーバを足すか）

サーバ側ストアは意図的に **opt-in かつ二次的**：

- **著作権の露出**。ROM/ディスクのバイトが我々のディスクに触れた瞬間、我々は NEC の
  BIOS と publisher のゲームを **保存**（不注意なら *配信* すら）することになる。ユー
  ザの IndexedDB に置けば我々は何も所持しない。サーバ層は「暗号文を1オーナー分だけ
  預かる、再配布しない」を保たねばならない — §7。
- **プライバシー**。ユーザのディスク library は「何を持っているか」の指紋。集めない
  のが最強のプライバシー保証。
- **コスト**。ディスクイメージは1枚数百 KB〜数 MB、library でユーザ当たり数十 MB。
  ブラウザに在るうちは我々には無料、我々のものになった瞬間に実ストレージ＋egress。

だからサーバストアを足すのは、IndexedDB が構造的に賄えない需要が出た時**だけ**：

1. **クロスデバイス** — 「PC で ROM を吸い出した、スマホでも使いたい」。IndexedDB は
   origin×ブラウザ単位で、端末をまたげない。これが認証層の主動機。
2. **大量／多ディスク管理** — 大きなゲーム library を、セッション・端末をまたいで整理
   して再選択したい。
3. **【要確認】共有／kiosk 端末** — ブラウザ単位ストレージが消去・共有され、ログイン
   済み identity だけが安定した鍵になる場合。

これらが噛まないなら、公開層だけで製品として完結する。

## 5. 認証層

### 5.1 認証モデル（gate としての volta）

`emulator.unlaxer.org` は独自認証を持たず、**volta-gateway の裏の保護 backend**。
volta 上では：

- 本番の `:80` は **volta-gateway**（traefik は遺物）。Cloudflare はワイルドカードな
  ので `emulator.unlaxer.org` は新規 DNS 無しで解決する。
- ルーティングは prod-yaml 直編集。`app_id` を指定した route は認証必須（gateway の
  `AuthChecked` state が volta `/auth/verify` を呼び、未認証は login へリダイレクト）。
  `public: true` の route は認証を素通り。我々が欲しいのは **前者** — エミュの route
  は `app_id` を持ち、public では *ない*：

  ```yaml
  routing:
    - host: emulator.unlaxer.org
      backend: http://localhost:<静的エミュのポート>
      app_id: app-emulator      # ⇒ 認証必須（public: true は付けない）
  ```

- volta の認証 spec（`auth-methods-landscape.md`）は **RP／クライアント側は完成**
  （エミュはその裏の保護アプリにすぎない）、**Device Authorization Grant
  (RFC 8628) / QR は Phase 1 で実装済** と報告している。これがクロスデバイス UX に
  効く（§5.3）。

エミュアプリ自体は認証非依存のまま：gateway から認証済みセッションを受け取り、
gateway が注入するヘッダからユーザ identity を読む（**【要確認】** 正確なヘッダ名／
クレーム — おそらく `X-Volta-*`。Phase 2 の配線前に gateway config で確認）。

### 5.2 per-user ストレージ

gate の裏で、各ユーザは volta identity で鍵付けされた自分専用の ROM フォルダを持つ：

- **ストア**：オブジェクトストレージ（ユーザ毎に1 prefix）または素の per-user ディレ
  クトリ。**【要確認】** volta が既にどちらを provision しているか。どちらも無ければ
  オブジェクトストレージが既定（安い・key 毎 ACL・FS quota で悩まない）。
- **レイアウト**：`romstore.js` に既にある `role` モデルを踏襲 — singleton（`rom`・
  `font`）は `{userId}/{role}`、多ディスク library は `{userId}/disk/{name}`（現状の
  単一 `disk` role の拡張）で鍵付け。
- UI が列挙するのは **メタデータのみ**（name・size・role・updated-at・content hash）。
  バイトは要求時に取得する。

### 5.3 アップロード／同期フロー

クライアントが権威のまま。IndexedDB が作業キャッシュ、サーバは同期ミラー。

```
ROM選択 ─▶ putRom(role, name, bytes)       // ローカル IndexedDB（不変）
              │
              └─▶ (認証層のみ) 「同期ON」なら ─▶ PUT /roms/{role}   // 暗号化
                                                  サーバストア

次回訪問 ─▶ restoreFromStore():
   ローカルhit? ─ yes ─▶ 即起動（高速パス, オフライン可）
                ─ no  ─▶ (認証層) GET /roms  → ダウンロード → putRom でローカル保存 → 起動
```

**クロスデバイスの筋書き**（この層の存在理由）：PC で ROM を選ぶ → 上へ同期される。
スマホでログイン（Phase 1 の Device Grant / QR でログイン自体が楽）→
`restoreFromStore()` はローカルに何も見つけず、サーバから取得し、IndexedDB にキャッ
シュして自動起動。**「一度アップ」がブラウザ単位ではなくアカウント単位になる。**

### 5.4 オーナー・プリシード（どこでもデモ）

サイトのオーナー（運営者＝あなた本人）も他のユーザと同じ一ユーザだが、追加能力を
一つ持つ：オーナーは自分の per-user フォルダに **full set の ROM ＋主要ディスク** を
*あらかじめ* **プリシード（pre-seed）** できる。目的は、外の見知らぬ端末でのデモを、
会場で毎回アップし直さず即起動させること。

- **仕組み**：オーナーは自分の **seed set**（full 起動 ROM・CGROM・主力 `.d88`
  library）を、*自分自身のアカウント* のサーバフォルダへ一度だけ置く — 通常の同期と
  同じ認証済みアップロード経路、または管理者側の一括 put で。実体は per-user ストアを
  事前に満たしただけ。
- **何が得か**：外部デモ（別会場・借り物ノート・スマホ）でオーナーがログイン →
  `restoreFromStore()` はローカルに何も無い → サーバの seed set を取得 → IndexedDB に
  キャッシュ → 自動起動。壇上でのファイル選択ゼロ。同じ端末の2回目は既にローカルに
  在る（オフライン可）。
- **著作権の境界 — 譲れない線**：seed set は **オーナー自身の吸い出しを、オーナー
  自身のアカウントに、オーナー自身が使うため** に置くもの。**配布ではない。** seed は
  **per-owner で、共有しない** — クロスユーザ seed も、訪問者に配る「既定 library」
  も、public-fetch も無い。オーナー鍵付け（§7）は通常のアップロード ROM と全く同じく
  seed にも適用され、seed は所有者以外のあらゆる identity から不可視。公開
  `3301.unlaxer.org` 層には seed は一切無い。

### 5.5 同期戦略：クライアント優先、サーバは任意

- **既定でクライアントが勝つ**。ローカル IndexedDB hit はネット往復無しで起動する。
  サーバを見に行くのはローカル miss（新しい端末／ブラウザ）か明示的な「今すぐ同期」の
  時だけ。
- **競合**：ROM は実質不変コンテンツ（ある吸い出しは変わらない）なので、content hash
  で鍵付けすれば大半の「競合」は no-op。同一 role で本当に異なるバイトなら、**明示的な
  上書き確認付きの last-write-wins** — 黙って潰さない。**【要確認】** ユーザが複数の
  起動 ROM を欲しがるか（欲しがるなら `role` に `disk` 同様の name 次元を足す）。
- **オフライン**：公開層の挙動を保つ — キャッシュが温まっていればアプリはサーバを一切
  必要としない。

## 6. 依存の向きと接続点

認証層の全体は **既存の継ぎ目一つ**：`romstore.js` にぶら下がる。その
`putRom / getRom / listRoms / clearRoms` インターフェースこそ、リモートストアが実装
すべき抽象そのもの。

```
machine.html ──使う──▶ romstore.js  (ローカル IndexedDB)        ← Phase 0, 不変
                            ▲
                            │  同じインターフェース
                     ┌──────┴───────────────────────────────┐
                     │ 同期層 (Phase 2, 新規, 認証層のみ)     │
                     │  ローカル IndexedDB  ⇄  リモートストア │
                     └───────────────────────────────────────┘
                                     │ HTTPS (gateway が認証)
                                     ▼
                          per-user サーバ ROM ストア
```

- **エミュコアには触らない**。`machine88.js` / `z80.js` / チップ群はホスティングを一切
  知らない。今と同じくバイトを受け取るだけ。
- **`machine.html` はほぼ不変**。`restoreFromStore()` と `refreshStore()` は既にストレ
  ージ API に集約されている。Phase 2 は同じシグネチャの *同期モジュール* を導入して
  ローカル IndexedDB とリモートストアをラップし、`machine.html` は `romstore.js` を直接
  ではなくそのラッパを呼ぶ。公開サイトではラッパはローカル専用に縮退する（リモート未
  設定）ので、単一コードパスが両ホストを賄う。
- **向きは一方向**：UI → 同期層 →（ローカルストア・リモートクライアント）。コアは上位
  に何も依存しない。リモートクライアントは identity のため gateway に依存するが、逆は
  無い。

## 7. セキュリティとプライバシー

ROM は **ユーザの著作権データを我々のインフラ上に預かるもの**。設計目標は、サーバを
足しても変わるのは *利便性* であって *露出* ではないこと。

- **アクセス制御**：全オブジェクトは volta identity にオーナー鍵付け。**共有・ユーザ横
  断列挙・public-fetch endpoint は無い** — バイトが出る唯一の道は、それを入れた同じ認証
  ユーザ本人だけ。これが「NEC/publisher の ROM を再配布しない」の具体的保証。**オーナー・
  プリシード（§5.4）が正当であり続けるのはまさにこれのおかげ**：seed set は他のオブジェ
  クトと同じくオーナー鍵付けなので、「オーナーの吸い出しをオーナーのデモに」が「公衆に
  配る library」に化けることは決して無い。
- **保管時暗号化**：サーバストアは暗号化。2水準あり、本物のトレードオフ：
  - *サーバ側暗号化*（我々が鍵を持つ）：最も単純、クロスデバイスは「ただ動く」。ただし
    我々は技術的にはバイトを読める。
  - *クライアント側暗号化*（**【要確認】** 価値があるか）：ブラウザがユーザ保持の鍵で
    アップロード前に暗号化し、我々は読めない暗号文を保存する — 「あなたの ROM を我々は
    持たない」の最強版。コスト：鍵を2台目の端末へ届けねばならない（volta ログインから
    導出？ passphrase？ Device Grant に載せた QR 鍵転送？）。非所持の約束をどれだけ強く
    したいかで決める。
- **通信**：HTTPS のみ（Cloudflare TLS）、かつ route は認証 gate の裏 — 未認証リクエスト
  はストアに届かない。
- **削除**：`clearRoms()` にはサーバ側の双子が要る — ユーザのオブジェクトを実削除し、
  「私を忘れて」で著作権バイトが実際に消えるように。
- **公開層はゼロ露出の性質をそのまま保つ**：そもそもサーバストアを持たないので、そこは
  何も後退しない。

## 8. 段階的計画

各フェーズは独立に出荷・検証できる。

**Phase 0 — 公開クライアント永続化。** ✅ *実装済。*
- 成果物：`3301.unlaxer.org` 静的・`romstore.js` IndexedDB・自動起動。
- 検証：ROM 選択 → リロード → 自動起動。DevTools でバイトが IndexedDB にのみ在り、ネッ
  トワークアップロードが無いこと。（＝現在の挙動）

**Phase 1 — volta 認証 gate。** *次。*
- 成果物：`emulator.unlaxer.org` ＝同じアプリを、`app_id` 付き route（`public` でない）で
  volta-gateway の裏に置く。ロードにログイン必須。**サーバ ROM ストレージはまだ無い** —
  ブラウザ単位 IndexedDB のまま、ただしログインの裏。認証の配線をストレージ作業から切り
  離し、Phase 2 が鍵付けに使う identity を確立する。
- 検証：未認証アクセス → volta login へリダイレクト。ログイン後はエミュがロードされ、
  公開サイトと全く同じ挙動。gateway がアプリの読めるユーザ identity を注入すること
  （**【要確認】** ヘッダ名）。

**Phase 2 — サーバ ROM 同期＋オーナー・プリシード。** *Phase 1 の後。*
- 成果物：per-user 暗号化ストア＋`romstore.js` をラップする同期モジュール、「同期ON」
  アップロード、クロスデバイスの miss 時ダウンロード、サーバ側削除、**オーナー・プリ
  シード**（§5.4）＝どこでもデモ用に、オーナー自身のフォルダを full ROM ＋主要ディスク
  set で事前充填する仕組み。
- 検証：端末 A で ROM 選択＋同期ON。端末 B（別ブラウザ）で同じユーザでログイン →
  `restoreFromStore()` がサーバから取得し、手動選択なしで自動起動。**プリシード検証**：
  オーナーフォルダを seed 済みにして *まっさら* な端末でログイン → seed set を取得 →
  ファイル選択ゼロで自動起動、2回目のロードはオフライン可。オーナー鍵付け（ユーザ B は
  ユーザ A のオブジェクト *や seed* を取得できない）、保管時暗号化、そして **公開サイトが
  無影響**（リモート未設定時に同期モジュールがローカル専用に縮退する）ことを確認。

## 9. 未解決の問い／リスク

- **【要確認】gateway の identity ヘッダ** — gateway が保護 backend に渡す正確なヘッダ／
  クレーム（per-user ストレージの鍵付けに要る）。記憶ではなく volta-gateway config を
  見る。
- **【要確認】ストレージ backing** — volta が既にオブジェクトストレージ／per-user dir を
  provision しているか、我々が立てるか。不明ならオブジェクトストレージを既定に。
- **【要確認】クライアント側 vs サーバ側暗号化** — 非所持の約束をどれだけ強くすべきか、
  クロスデバイスで鍵はどこに住むか。
- **`role` モデルの拡張** — 多ディスク library は `role` に name 次元を持たせる必要が
  ある（現状 `disk` は singleton）。`romstore.js` で安く拡張できる。ローカル・リモート
  両用に一度でやる。
- **認証 spec のドリフト** — 古いメモは volta の OP（発行側）を空と記録していたが、現行
  spec は RP 側完成・Device Grant は Phase 1 と報告している。我々が依存する RP／保護アプリ
  経路は確定した方だが、配線前に **生きた spec で確認** すること。
